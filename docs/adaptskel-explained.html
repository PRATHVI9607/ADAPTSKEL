ADAPTSKEL, explained from zero

A fully-dynamic shortest-path engine, every concept defined from scratch — graphs, Dijkstra, Link-Cut Trees, the OMv wall, and congestion control. Built for the student who has never seen any of this before. Catch each entry as you master it.

#001

The Problem

What ADAPTSKEL actually solves, and why the obvious solution is too slow.

1.1 What is a graph? Definition

A graph is "dots connected by lines." Formally G = (V, E):

  • Vertex (or node): a dot. Here, each vertex is a city — Delhi, Mumbai, Bangalore.
  • Edge: a line joining two vertices. Here, an edge is a fiber-optic cable between two cities.
  • Weight: a number on an edge saying how costly it is to cross. Here it is latency — milliseconds a signal takes (we use 1 ms per 100 km).

Our backbone: 34 cities (vertices), ~60 cables (edges). It is undirected — a cable carries traffic both ways.

1.2 What is a shortest path? Definition

A path from A to B is a sequence of edges you follow. Its length is the sum of edge weights. The shortest path has the smallest total. The shortest-path distance δ(A,B) is that smallest total — the core quantity of the whole project.

Worked example

Delhi→Mumbai direct = 14 ms. But Delhi→Jaipur→Mumbai = 6 + 7 = 13 ms. The 2-hop route is shorter even with more hops. So δ(Delhi,Mumbai)=13.

1.3 What is SSSP? Definition

SSSP = Single-Source Shortest Paths. Fix ONE start vertex (the source) and find the shortest distance to every other vertex. Our source is Delhi (node 0): "from Delhi, shortest latency to all 33 other cities?" (APSP = all-pairs, far more expensive — not our main job.)

1.4 "Dynamic" and "fully dynamic" Definition

A static graph never changes. A dynamic graph changes: insert an edge (new/recovered cable), delete an edge (fiber cut), reweight an edge (congestion). Terminology:

  • Incremental — only insertions.
  • Decremental — only deletions.
  • Fully dynamic — both, interleaved, forever. This is us.

1.5 Why is this hard?

Obvious fix: after every change, recompute everything with Dijkstra — costing about O(E log V). On 100,000 edges that's huge work per change. If the network changes thousands of times a second, from-scratch recompute is far too slow. The dream: when one edge changes, do work proportional only to the part that actually changed. That's what ADAPTSKEL chases.

Background Concepts

Every idea you need before the algorithm makes sense — defined plainly.

2.1 Big-O notation Theory

Big-O describes how running time grows as input grows, ignoring constants. "Double the graph — what happens to time?"

NotationNameMeaningn = 1,000,000
O(1)constantwork independent of size (a table lookup)1
O(log n)logarithmicgrows very slowly (halvings to reach 1)≈ 20
O(log² n)log-squared(log n) × (log n)≈ 400

O(E log V)Dijkstragrows with edge count — the expensive onelarge

Goal: keep queries near O(1)/O(log n) instead of the O(E log V) of a full recompute.

2.2 Amortized complexity Theory

Some operations are occasionally expensive but usually cheap. Amortized cost = total cost of many operations ÷ number of operations. Like a water heater: most showers the water is already hot; rarely it reheats the tank; averaged over a month, cheap per shower. "O(log² n) amortized" means on average across many updates, not every single one.

2.3 Dijkstra's algorithm Algorithm

The classic shortest-path method. Picture water spreading from the source — near cities first, far later:

  1. Tentative distance = 0 for source, ∞ for everyone else.
  2. Keep a priority queue of vertices ordered by tentative distance, smallest first.
  3. Take the smallest vertex x; its distance is now final. For each neighbor y: if dist[x] + weight(x,y) < dist[y], lower dist[y]. This is relaxation.
  4. Repeat until the queue empties.

Cost O(E log V). Relaxation — "found a cheaper way to reach y, so lower its distance" — is the single most important sub-idea. ADAPTSKEL's updates are "do only the relaxations actually needed."

2.4 Trees, forests, spanning trees Definition

  • Tree: connected, no cycles — exactly one path between any two vertices. k vertices → k−1 edges.
  • Forest: several separate trees.
  • Spanning tree: a tree using a subset of edges that touches all vertices — the loop-free backbone connecting everyone.
  • Spanning forest: one spanning tree per disconnected piece.

Why care: ADAPTSKEL keeps a spanning forest (F₁) as a fast skeleton. Trees are cheap to query; the full cyclic graph is not.

2.5 Splay trees Data structure

A binary search tree finds/inserts/deletes in O(log n) if balanced. A splay tree self-balances by rotating every touched node to the top, so hot items drift up — giving O(log n) amortized. It's the engine inside the Link-Cut Tree; you needn't master it, just know "balanced-tree machinery."

2.6 Heap / priority queue Data structure

A container where you add items with a priority and quickly remove the smallest. A heap gives O(log n) add and remove-min. Dijkstra uses it to always grab the closest unprocessed vertex; our code uses Python's heapq.

2.7 Link-Cut Tree (LCT) Data structure

Maintains a forest with all of these in O(log n) amortized: link(u,v) join two trees, cut(u,v) split, connected(u,v), and path queries (total weight u→v). "A spanning forest you can rewire fast and ask path questions about." Holds F₁. File: core/python/lct.py.

2.8 Euler Tour Tree (ETT) Data structure

Another forest structure, great at edits, connectivity, and — crucially — finding a replacement edge when a tree edge dies. It writes each tree as a linear "Euler tour" (walk around the tree, list each vertex you pass) stored in a balanced tree, so edits are O(log n). Holds F₂. File: core/python/ett.py.

2.9 The Holm "level" trick Theory

When you delete a skeleton edge, the tree splits and you must search leftover edges for one that reconnects the halves. Naively slow. Holm, de Lichtenberg & Thorup (2001) give each edge a level (small integer); edges get promoted as they survive searches; searching high→low with only-promotions bounds the total lifetime work → O(log² n) amortized reconnection. Just know: "levels make replacement search provably cheap over time."

2.10 Ramalingam–Reps Algorithm

Ramalingam & Reps (1996) — the standard way to update shortest paths incrementally, touching only vertices whose distance actually changes.

  • Decrease (edge cheaper/new): distances only drop. Seed relaxation at the changed edge; it ripples out and stops when nothing improves.
  • Increase (edge costlier/deleted): distances only rise, and only for vertices whose shortest path used that edge. (1) find the affected set — vertices that lost their support (a vertex is supported if some neighbor still gives dist[nbr]+w = dist[v]); (2) reset them to ∞; (3) recompute with a small Dijkstra seeded from unaffected neighbors.

Exactly what the honesty fix implements. "Output-sensitive" = cost proportional to what changed, not the whole graph.

2.11 The OMv conjecture Theory

OMv = Online Matrix-Vector multiplication. You get an n×n 0/1 matrix M to preprocess. Then 0/1 vectors v₁…vₙ arrive one at a time; you must output M·vᵢ before seeing the next vector. The conjecture: you cannot beat ~O(n²) per vector (no algorithm does all n rounds in O(n³⁻ᵉ)). Unproven, but widely believed.

Why this matters to you

If OMv is true, then exact fully-dynamic shortest paths cannot have BOTH fast updates and fast queries — a hard wall. So the old README claim "O(log² n) exact fully-dynamic SSSP" claimed to break a wall that (under a believed conjecture) can't be broken. We rewrote it to be honest. Stating this correctly impresses an examiner — you know your problem's limits. A conjecture = a precise statement believed true with strong evidence but not yet proven (like P ≠ NP).

#004

The ADAPTSKEL Algorithm

The heart, in finest detail. Name = Adaptive Skeleton.

4.1 Big picture — two layers

Every edge lives in exactly one layer:

  • F₁ Skeleton — the hot edges (often on asked-about paths), stored in the Link-Cut Tree. Kept as a spanning forest (connects all, no cycles) → fast O(log n) questions.
  • F₂ Residual — the cold edges, stored in the Euler Tour Tree with Holm levels. Still real edges; useful as replacements later.
Invariant (a rule that must ALWAYS hold)

Every edge is in exactly one of F₁ or F₂, and together they are the whole edge set: F₁ ∪ F₂ = E.

4.2 The three novel ideas

Idea 1 — Heat-based promotion. Each edge has a heat score = how many of the last W queries used it. Cross threshold T → promote F₂→F₁. Fall back → demote F₁→F₂. The skeleton auto-crystallizes around the busy subgraph. File: heat_table.py.

Idea 2 — Delta-LDB batching. Distance events split in two:

  • DECREASE (insert/cheapen) → distances shrink. A stored value that's slightly too big is a safe upper bound → can be deferred/batched.
  • INCREASE (delete/raise) → distances grow. A stored value too small is wrong → must be fixed before any query.

"Decreases lazy, increases urgent" is what bounds the work. File: delta_ldb.py.

Idea 3 — Two-level forest with fast repair. Delete a skeleton edge → tree splits → find a replacement from F₂ to re-stitch (Holm levels make it provably cheap).

4.3 What ADAPTSKEL stores

FieldMeaning
_dist[v]shortest distance source→v. The answer table. Kept exact, incrementally.
_pred[v]parent of v on its shortest path — lets us rebuild the actual route.
_adj[u][v]neighbors + weights — the real weighted graph.

4.4 INSERT — step by step Algorithm

  1. Ensure both vertices exist; add edge to the real graph.
  2. Repair distances (DECREASE). A new/cheaper edge only shrinks distances → run _relax_decrease: if dist[u]+w < dist[v], set dist[v]=dist[u]+w, record predecessor, push into a queue; then Dijkstra-relax outward, stopping when nothing improves. Only affected vertices touched.
  3. Choose layer. Hot & not already connected in skeleton → F₁ (link in LCT). Else → F₂ (add to ETT).
Battle example — insert Jaipur–Mumbai (w = 7)

Source Delhi, dist[Jaipur]=6, dist[Mumbai]=20 (long detour). Check 6+7=13 < 20 → Mumbai improves to 13, Jaipur becomes its predecessor. Then relax Mumbai's neighbors (Pune 25 → 13+8=21)… only cities that actually get closer update. Delhi, Jaipur, far untouched cities never move.

4.5 DELETE — step by step (the hard one) Algorithm

  1. Remove edge from the real graph.
  2. Remove from its layer. If F₂ — just drop. If F₁ — the tree splits → _f1_deletion_repair: search F₂ (Holm levels, high→low) for a replacement that reconnects the halves; move it up to F₁.
  3. Repair distances (INCREASE) via _recompute_increase (Ramalingam–Reps):
Phase 1 — find affected set

From u,v, find vertices that lost support. x is supported if a neighbor y still gives dist[y]+w(x,y)=dist[x]. No support → affected (its old distance is now too small). Spread: a vertex fed through an affected vertex with no other support is affected too.

Phase 2 — recompute

Set affected vertices to ∞. For each, take the best dist[unaffected neighbor]+w as a fresh estimate; run a small Dijkstra among the affected. Unaffected vertices never touched.

Battle example — cut Jaipur–Mumbai

Mumbai was Delhi→Jaipur→Mumbai = 13; Pune hung off Mumbai at 21. Cut the cable. Phase 1: does Mumbai still have support? If the best alternative is …→Hyderabad→Mumbai = 18, Mumbai is supported (worse value); only Mumbai + Pune need updating. Phase 2 lifts Mumbai→18, Pune→26. The rest of the country untouched.

4.6 QUERY — step by step Algorithm

  • s = source (Delhi): answer already in _dist[t] → return it, O(1). No Dijkstra, no oracle. The honest speed win.
  • s ≠ source: structure only maintains the fixed source, so run one honest Dijkstra from s (O(E log V)) — no free lunch, stated plainly.

Then update heat: bump every edge on the returned path; cross T → promote; gone cold → demote. That's how the skeleton adapts to usage.

4.7 The five invariants Definition

  1. F₁ ∪ F₂ = E — every edge in exactly one layer.
  2. F₁ is a valid spanning forest — connects what it should, no cycles.
  3. dist[v] exactly correct — checked vs the NetworkX oracle. The big one: proves incremental maintenance equals from-scratch, every time.
  4. heat[e] ≤ W — no heat exceeds the window.
  5. F₁ edges are hot (heat ≥ T/2) — skeleton really holds hot edges, with hysteresis.
Hysteresis

Promote at T, demote only at T/2. The gap stops an edge "flickering" in/out when its heat hovers near the line — same reason a thermostat doesn't switch on/off every second.

4.8 The honest complexity — say exactly this Read before demo

  • Source-rooted query: O(1) — genuine table read.
  • Insert / delete: output-sensitive — work proportional to affected vertices.
  • Exact — oracle-verified after every mutation.
  • NOT worst-case polylog for updates (OMv wall). Updates beat a Dijkstra rerun when changes are local; worst case ≈ one Dijkstra.
#005

Congestion Control

"Many people down the same road" — detected, and solved by rerouting.

5.1 The problem Congestion

  • Capacity — max traffic a cable carries (Gbps). Hubs 100, others 40.
  • Load — traffic actually flowing now.
  • Utilisation = load / capacity. 0.5 half-full; 1.0 full; >1.0 overloaded → excess dropped (lost).

The plain router sends every flow down the lowest-latency path, ignoring load. Many big flows pile onto the same hub cable → its load exceeds capacity → congestion — while other slightly-longer cables sit empty.

5.2 The solution — load-adaptive weights Congestion

Make a cable's effective cost rise as it fills, so later flows pick emptier detours. This is a stream of edge-reweight events — exactly what ADAPTSKEL absorbs. Effective weight (file routing/congestion.py):

effective_weight = base_latency × (1 + alpha × over^beta) × heat_term
where  over = max(0, utilisation − knee)
PieceDefaultMeaning
base_latency—the cable's normal latency (raw weight)
knee0.8below 80% full, penalty = 0 → weight = base latency → routes identically to baseline (never worse)
alpha10how big the penalty gets near full (~11× cost)
beta3how sharp the ramp — stays ~0 until near the knee, then shoots up
heat_termγ=0.5heat-as-load: only active above the knee; busiest-AND-hottest links shed first
The guarantee (proven by a test)

Below the knee, smart weights equal baseline weights exactly → identical routes when nothing is congested → congestion-aware loss is ≤ baseline, always. It only diverts under real pressure.

5.3 The loss model Congestion

Measured on the whole demand set together (not one at a time):

  1. Route every demand, accumulating load per cable.
  2. Per cable: overload = max(0, load − capacity).
  3. total lost = (demands with no path) + Σ overloads.
  4. loss% = total lost / total offered × 100.

This aggregate view reveals congestion the old per-demand check missed.

5.4 Calibration Congestion

The traffic generator offers far more than the backbone can carry, so raw numbers = 100% loss for everyone (meaningless). Fix: uniformly scale demand so the healthy network's busiest cable sits at ~85% (target_max_util = 0.85) — "an ISP run at a sensible load." Calibrate once on the healthy net, hold it fixed as cables fail → failures create real overload → the win becomes visible.

5.5 The result (measured live) Result

ScenarioLatency-only lossCongestion-aware lossWin
Healthy0%0%0
Cut 4 hub cables71.5%27.2%+44 pp

Busiest-link oversubscription also drops (e.g. 7.0× → 3.6×): traffic really is being spread. ("pp" = percentage points.)

#006

Methodology & How It's Verified

How the project was built and proven correct.

6.1 Reference-first

Write the algorithm first in clean, readable Python whose only job is to be correct and understandable. Optimize/port to C++ only after correctness is nailed. We're at the correct-reference stage; the C++ port is future work.

6.2 Oracle testing Definition

An oracle is a separate, trusted source of correct answers — here, NetworkX's Dijkstra. The test (tests/test_correctness.py):

  1. Generate thousands of random inserts/deletes/queries.
  2. Feed the same sequence to ADAPTSKEL and the oracle.
  3. After every query, assert both distances are identical.

Passes on 200 vertices × 2000 ops across many seeds — strong evidence the maintenance is exact. 76 tests total, all green. This style — assert a property ("always equals the oracle") over random inputs — is called property-based testing.

#007

Results & Discussion

What's proven, what's honest, what's next.

7.1 Correctness (the solid result) Result

ADAPTSKEL's maintained distances match the oracle exactly on every query across thousands of random ops and seeds — including strict debug mode checking all five invariants after every mutation. Defensible claim: exact fully-dynamic SSSP labels, oracle-verified.

7.2 Query performance Result

Source-rooted queries are genuine O(1) — a label read, no recompute, no oracle. Arbitrary-pair queries honestly cost one Dijkstra.

7.3 Congestion Result

Healthy = 0% both; under 4 hub failures, loss cut 71.5% → 27.2% (+44 pp), provably never worse. Busiest-link oversubscription 7.0× → 3.6×.

7.4 Honest limitations (state these — they earn credibility) Honesty

  • Not worst-case polylog for updates (OMv). We deliver output-sensitive updates + O(1) source query, and say so.
  • Fixed single source — arbitrary-pair queries get no speed-up.
  • Python reference, not the C++ performance port yet.
  • Congestion control is single-path rerouting, not yet multi-path (ECMP) splitting.

7.5 Honest comparison

ApproachSource queryUpdateExact?
Dijkstra rerunO(1) after an O(E log V) recompute every changeO(E log V)✓
ADAPTSKELO(1) alwaysoutput-sensitive (≤ one Dijkstra)✓ oracle-verified

ADAPTSKEL wins when changes are local — one cable fails, only a small region shifts, so far less than a full recompute, while source queries stay instant.

7.6 Future work

  1. C++ performance port (realize the constant-factor speed the design allows).
  2. Multi-path / ECMP congestion splitting.
  3. Extend fast queries beyond the single fixed source.
